Hitbox

  • Language
  • Watch
  • Edit

A hitbox defines the physical "boundaries" (or an approximation thereof) of a block or entity. Hitboxes are utilised in the calculations of collisions and targeting.

Unlike with block models, all hitboxes are hardcoded and cannot be changed without modification, even via data packs.

Entities hitboxes are color-coded, and can be seen through the F3 + B debug hitboxes.

Types of hitbox Edit

Entity hitboxes Edit

Bounding box Edit

An entity's bounding box manages collisions with other entities and blocks. It is also used for detecting hits and interactions from the player's crosshair or other entities (with an exception to the ender dragon).

The bounding box moves with the entity but maintains a static orientation without rotating.

The bounding box can be viewed via the F3 + B debug hitboxes shortcut.‌[Java Edition only] The box is shown as white outlines around the entities.

Entities bounding box can change its shape depending on the situation. For example, a pufferfish when approached expands its shape, going from unpuffed, to semi-puffed, then to fully puffed. It instantly shrinks its shape back to normal once it cools down or no longer takes any kind of damage. The player can also change its bounding box by sneaking, crawling, swimming, using elytra, or any other various means.

Eye height Edit

Eye height is a flat red hitbox that typically appears on the entities' eyes or head. This hitbox is used for detecting when an entity is submerged in water or phasing through a solid block. In this state, the game usually times the entity running out of air or suffocate.

Eye height is also used to determine if two entities can see and follow each others by their eye contact.[1]

When the debug hitboxes setting is enabled, a long blue tint appears and placed in-line with the eye height hitbox. This blue tint shows which direction an entity is looking, as its rotation changes based on the entity's head rotation.

Passenger attachment point Edit

Passenger attachment point is a small green hitbox that shows the point where the entity rider is attached on the entity vehicle. This hitbox appears on a vehicle entity only when it is mounted by another entity.

When an entity rides another entity, the rider's bounding box aligns precisely with the bottom side of the passenger attachment point hitbox.

Block hitboxes Edit

Collision box Edit

Collision box manages block collision with entities.

Collision box is not present for all blocks, as all non-solid blocks such as grass, sign, button, lever and torch have no collision box.

Some blocks may have custom collision box shape, such as slabs, fences, and stairs. Some also have custom collision behavior that can make them semi-solid, such examples are scaffolding and hanging sign.

Blocks may have different collision box than the interaction box, for example, the lectern having the head almost non-collideable, despite the black outline showing so, and the fences having one and a half blocks tall collision box than the interaction box.

Interaction box Edit

Interaction box manages block interactions, such as block placement, destruction, right-click, and other interactions coming from the player's crosshair. Interaction box is shown when looking at a given block, as it becomes a black outline when the player crosshair is pointing at the block.

Blocks interaction box are usually contained within its own 1x1x1 cube space, but some blocks are capable of extending outside of its own space.[2] There are several instances where this can be seen:

  • The arm of an extended piston – specifically the part that protrudes into the piston's block space.[2][3]
  • The extended top of an opened shulker box.[2]
  • The very top part of a lectern's head.[2] This is especially obvious when a block is placed above the lectern, making lectern's outline overlap with the block.

This behavior causes the ray-casting that usually detect the interaction box precense to never collide with it, even though it should.[2]

Miscellaneous hitboxes Edit

Third-person view camera collision box Edit

Third-person view camera has a collision box that can collide with solid blocks. It does not collide with transparent blocks, such as glasses and glass panes, and non-solid blocks. However, the third-person view camera does collide with the barrier block.

Smoke particles collision box Edit

Smoke particles have a collision box that can collide with solid blocks. Smoke particles come from the campfire, and unlike with most particles, which does not collide with anything. Like the third-person view camera collision, it does not collide with transparent blocks and non-solid blocks, and collides with the barrier block.

The hitboxes of all entities are mostly shaped in cuboids and do not always overlap with the model of the entity. For example, the witch's hitbox does not include the top section of its hat, and the wither's hitbox does not include the two heads on its sides. Because most hitboxes are cuboids, it is standard to measure them by their width and height, with both width and height being measure in blocks tall (so a width of 0.6 means the entity is 0.6 blocks wide).

Do note that not every entities has cuboid hitbox, like the ender dragon. This list only measure their bounding box.

Ender dragon Edit

Ender dragon is composed of nine unique hitboxes. Ender dragon has a single big white bounding box, and eight small green hitboxes inside, which players can deal damage to. These green hitboxes move interchangeably with the ender dragon moving and rotating to a specific location. Like the bounding box, the green hitboxes does not rotate.

Variable hitbox sizes Edit

The size of an entity's hitbox is not necessarily static and may change depending on certain circumstances. For instance, a player's hitbox gets smaller while crouching, and the hitbox of a pufferfish changes while it's puffed out.
