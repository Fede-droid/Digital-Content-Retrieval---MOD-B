A need for a new system for MCW interwiki links

Given the discussion about adding more themes, custom themes and CSS on spanish wiki navboxes, and updating interwiki links for navboxes and other templates. The discussion has turned towards a pressing need for a new system for links between templates and articles between all language wikis for MCW.

The problem Edit

The current interwiki system relies on local links on every single article on every single language. This means that in order to link between each language wiki's article, links must be added individually to every page on every wiki. For links between templates and modules, each documentation page must have 2 sets of interwiki links for both the documentation subpage and the base template page across all languages.

This causes many problems since editors must update the same page on 8 different languages with the same interwiki link 8 times over. If it would take only 20 seconds to update 1 interwiki link in one place, it currently takes 4 minutes to update the same link on 8 different pages for all languages. If the LZH, Japanese, Dutch, and Thai wikis fork from fandom, it would take even longer to update interwiki links locally since there's now 13 languages instead of 9 (including EN).

Having 9 different copies of the same set of interwiki links on 9 different wikis is very problematic. It makes it hard for editors to keep interwiki links updated, meaning some wikis have dead, outdated, or missing interwiki links. This can make it difficult for readers to find the language wikis from various articles and attract new readers and editors to the language wikis.

Searching for a solution Edit

A new system for interwiki links would provide single place for all languages to add an interwiki link. This would mean that editors would not have to manually update the same interwiki links 12 times on 12 different langauge pages to link to one page on a particular language, meaning that it would provide numerous benefits for the Minecraft wiki:

  • Reduced maintainance load for editors, allowing editors to focus on other tasks such as improving content/design of the wiki, if the process for editors to update interwiki links is centralized. All the current maintainance problems of the current system would go away.
  • Less clutter in Recent Changes, if interwiki links are updated in a centralized place or by bots.
  • Implementation of the new interwiki link system before the 4 dev wikis go live mean it will be much easier to integrate the newly forked languages with existing forked languages.
  • A new system would provide the new wiki with a numerous benefit over Fandom. An ability to deploy interwiki links across multiple articles without added manual effort, means it's much easier for readers to access various languages of the wiki. Language wikis will have a much easier time finding new readers and editors.

In the discord discussion, it was mentioned the Wikipedia system for interwiki links was not favorable for MCW. Some suggestions included a centralized system either on the EN wiki, or another off site place where bots could automatically deploy interwiki links for all languages.

Discussion Edit

I am in favor of an off site system for interwikis, away from the EN wiki for the following reasons.

  1. In some instances, there may be a page or template with equivalent pages between multiple languages, but not on EN. For example, both PT and FR wikis have Template:Community, but EN wiki has no community navbox template.
  2. A use of an off site centralized system for interwiki links away from EN wiki means the process to update an interwiki link on EN is exactly the same as with other language wikis, meaning one only needs to remember a single set of instructions for updating interwiki likes, whether thats EN or another language.

I have joined the MCW board server, so if someone knows some pointers on how the new Minecraft Wiki can get a new interwiki system set up and deployed (hopefully before any of the 4 dev staging wikis go live), Let me know. MarkusRost noted that a centralized system could break interwikis for split vs. grouped pages, since some language wikis have merged pages and others have split pages. I would assume that even with a centralized interwiki system, manual interwiki links probably would still work. On Wikipedia, it is possible to enter a manual interwiki link and it seems to override the default interwiki link. A centralized interwiki system does not mean that all manual interwiki links need to be removed. A few exceptions may still exist for a few pages or so, and they can still use manual interwiki links as needed for those far edge cases. However, in most cases, almost all pages should move to the centralized interwiki system. Delvin4519 (talk) 02:53, 9 February 2024 (UTC)Reply[reply]
One problem with using bots to manage interwiki links, is the fact that using bots to manage interwiki links still does not change the fact that there would still be up to 182 interwiki links (13 * 14) for each set of pages to manage across 14 different wikis (1 EN + 9 current + 4 future = 14). While the use of bots obviously can remove a decent amount of human manual updates of interwiki links, there would still be a need to run 14 different scripts on 14 different wikis. Bots can sometimes mistakenly delete links, and then another bot script run on another wiki would need to re-add those deleted links. This would still cause issues with these bot runs that can clog up page revision histories if one bot run deletes an interwiki link and a second bot run re-adds the deleted interwiki link (2 entries in revision history), when it only needs to be updated once.
Now if the 182 interwiki links could be brought down to to just 29 (14 * 2 + 1), that could of course could remove a lot of maintainance overhead and lower the need for bots to manage these interwiki links. Having 182 interwiki links sprawled across 14 different wikis, as oppossed to 29, is obviously going to require more maintainance overhead than is warrented, since wikis should focus on improving themselves, and that should be the goal right? To focus on improving our own wikis and not wasting time managing 182 interwiki links. As such, I would only consider bots to only be partial fix to the problem, although it is most certainly better than nothing, which can't be understated. However, it probably isn't sufficent enough long-term, once the 4 pending wikis join the fork. Delvin4519 (talk) 03:10, 10 February 2024 (UTC)Reply[reply]
