Better navbox and infobox naming

The following discussion is closed. Please do not modify it.

The new navbox and infobox naming scheme is in the process of being implemented. - Harristic / Talk 19:07, 25 May 2024 (UTC)

This is a proposal to make three changes to how we name navbox and infobox templates:

  1. Prefix all infobox and navbox template names with "Infobox" and "Navbox" respectively
  2. For spin-off navboxes, remove "Minecraft" from the navbox name
  3. For spin-off infoboxes, space out the words in the name (this probably sounds confusing, the example below should help)

Examples for how names would change if this proposal passes:

  1. Item would be renamed to "Infobox item", Items would be renamed to "Navbox items"
  2. Minecraft Dungeons entities would be renamed to "Navbox Dungeons entities"
  3. DungeonsCosmetic would be renamed to "Infobox Dungeons cosmetic"

Reasoning for the three parts of the proposal:

  1. This simply makes it much clearer what a template is and makes everything much neater in my opinion, currently "Item" and "Items" are entirely different templates despite the only difference between the names being that one is singular and one is plural.
  2. "Minecraft" being in the name essentially just acts as fluff, it lengthens the name while having no reason to be there because "Dungeons" or "Earth" already clearly signifies the spin-off the navbox is focused on. Removing "Minecraft" from the name also counteracts the added length from adding "Navbox" to the name.
  3. This is a very small change but I wanted to mention it anyway, it's just always bugged me that spin off infoboxes have names like "DungeonsItem" or "LegendsEntity", I'd rather they be named the same way that navboxes are for consistency and neatness.

One opposing reason for this could be that template names would become too long, so these are my thoughts about that/counterargument to that:

  1. Spin-off navboxes could be made "too long" if "Navbox" were prefixed on all of them, however point two of my proposal counteracts this. By removing "Minecraft" and adding "Navbox", these template names actually become shorter funnily enough.
  2. Infobox templates are only used once per page, therefore typing out a template with a slightly longer name shouldn't be an issue. There can sometimes be multiple navboxes on a page, but the same sentiment applies, these templates are used so little times on any given page that the extra time it takes to type "Navbox" or "Infobox" is negligable.
  3. There are indeed some cases where template names would be almost comically long if this proposal passed, such as Template:Bedrock Dedicated Server versions, however these are fringe cases and will be uncommon. Frankly I don't think having a very long name in these rare cases is bad, especially since all version pages will only ever have one navbox, unlike other mainspace pages which can often have multiple. However, if this is considered a big deal, we could certainly create shortcut redirects for these navboxes, or just shorten the names case by case. The example I gave could be renamed to "Navbox BDS versions" as an example, these case by case shortenings wouldn't really need large discussions before their implementation.

Another opposing reason could be that Template:Navbox see also would become far too large, so these are my thoughts about that/counterargument to that:

  1. We can use piped links to make it so that, as an example, "Navbox Dungeons items" just displays as "Dungeons items".
  2. My first point actually doesn't work, because Navbox see also displays all the links in code and surrounded with curly brackets (or whatever they're called). This means that anyone looking at Navbox see also will think that typing {{Dungeons items}} would be the correct way of adding the navbox, which it wouldn't be. To counteract this issue I suggest we simply remove the code and curly brackets surrounding every link, Navbox see also acting as a tutorial on how to add a template is super random and unnecessary, and the code and curly brackets seemingly make the navbox a lot larger than it needs to be anyway.

Two things to note:

  1. Thematic navboxes already have the prefix naming scheme because a smaller group on Discord agreed on that happening, however I discovered it was controversial to apply those changes to other new navboxes, so I've created this proposal (and expanded its scope).
  2. A bot would need to implement these changes.

It's not just about breaking older revisions, there's also the HUGE issue of all the interwiki links pointing to the English wiki navboxes will all completely break. Especially with things like the /doc subpage that lists all interwikis for both the base template and the doc subpage, if redirects don't exist for the /doc subpage and trying to correct the interwiki links. A bot who has bot permissions on the English wiki may not have bot permissions on all the interwikis, same for admin, etc.. There are 80 general navboxes, and 8 interwikis that could become 12 interwikis after NL, JA, TH, and LZH fork, with 2 interwikis for each navbox (main page and /doc subpage). We're talking about over 1,000 - 1,600 interwiki links to our navboxes that will all need updating (not all interwikis have all our navboxes), and it's not so straightforward since it's 12 different wikis with the bot permissions across different wikis issue. Since navboxes are supposed to provide quick and easy navigation, it's important for navboxes to have functional interwikis. When the German wiki renamed their navboxes and /doc subpages, a handful of interwikis were left behind broken on the English wiki. Delvin4519 (talk) 12:41, 3 February 2024 (UTC)
The solution seems incredibly clear to me. Couldn't whoever is doing the edits just contact each language wiki and request the bot role for their bot account? If for whatever reason that is not an option, someone could just do the bot edits with their normal account. If I'm not mistaken the main point of the bot role is to prevent recent changes from being utterly unusable, but each language will only need like 200-300 edits, which is a relatively small amount. - Harristic / Talk 13:09, 3 February 2024 (UTC)

Per another discussion in discord. It's been raised that renaming all the navboxes to be prefixed with "Navbox" with a spacebar following it before the navboxes' names themselves, causes all search results for Template:{{Navbox}} to completely break. This issue can be seen with "Template:Navbox see also", but since only a single navbox is prefixed with it, means it is still possible to filter the results out to find any missing or unaccounted for navbox using Template:Navbox via the search function. Special:WhatLinksHere has issues with missing navboxes hidden behind LoadBoxes or <includeonly> tags, so a search function is needed to find any navbox that uses Template:Navbox, or is not accounted for within categories/documentation/interwikis.
Regularly searching for templates using Template:Navbox remains an extremely useful function to have as it allows for linking via interwikis. Even recently, there were 2 navboxes unaccounted for on the English wiki, and more on interwikis, with some language wikis having 1 or 2 navboxes that used Template:Navbox instead of SimpleNavbox.
I am not sure if I would be in favor of deleting Template:Navbox to make way for renaming navboxes to be prefixed from "Template:Something" to "Template:Navbox something something". Template:Navbox still has usages on English wiki, as well as on DE and RU wikis. For as long as Template:Navbox continues to exist on the English wiki and on the language interwikis, a way to search for Template:Navbox is still an essential function to have.
Perhaps another solution is necessary for this problem, one that does not involve deleting Template:Navbox. Special:WhatLinksHere has limitations with showing all transclusions, and failing to show usages behind includeonly tags or loadboxes. Delvin4519 (talk) 01:16, 5 February 2024 (UTC)
