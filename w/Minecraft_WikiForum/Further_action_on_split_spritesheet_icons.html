Further action on split spritesheet icons

In this topic we’re discussing renaming individual icon files to not refer to the now-abolished spritesheets, potential full replacement of Module:SpriteFile with direct usage of said icon files, dealing with {{Schematic}} icons, and replacing inventory icons for items with smaller ItemSprite files.

Foreword Edit

Soon after forking, we’ve went ahead with what I, BabylonAS, have called “Operation Sprite Ender” (in reference to the Texture Ender tool for converting outdated texture packs): replacing large spritesheets used ever since early 2011 with individual files. Back in the day, spritesheets (or texture atlases) were necessary because browsers couldn’t properly handle large numbers of requests that using individual files would incur. However, with the advent of HTTP/2, standardized in 2015, and recently also HTTP/3, standardized in 2022, as well as their adoption by browsers, this mostly became a non-issue as such versions of HTTP are more well-suited for handing requests in parallel. Using a large spritesheet instead of individual smaller files also resulted in unnecessary bandwidth consumption as the entire file is loaded, only for a smaller portion of it to be used on a page, whereas with individual files, only those that are actually used are, obviously, loaded. Finally, this also made our articles differ more from similar ones on the old Fandom wiki, boosting our search ratings which is critical for our current SEO competition with our own “shadow”.

For now, however, the now-split icons are in some sort of intermediate stage. We have Module:SpriteFile created as a drop-in replacement for Module:Sprite, sharing the latter's usage syntax but supplying individual files instead of a spritesheet. The files themselves are named File:spritesheetSprite id.png, bearing the traces of their origin. However, there is no more need to abide by the older practices. Our new friends at RuneScape Wiki use a single, simple template to add “fancy” links with icons, which were the largest former use case for sprites on Minecraft Wiki outside of Inventory slot and schematic templates ({{BlockGrid}}, {{Layered blueprint}}). Our icon link templates, however, still rely on SpriteFile's backward compatibility with the old sprite technology.

Finally, {{Schematic}}, widely used on redstone circuit articles, is still using Module:Sprite. That specific spritesheet has not been split yet because its text identifiers have problematic characters such as the forward slash, which is the directory separator on Unix-like systems. Furthermore, some of its sprites are duplicates of block icons. Two more relatively small spritesheets are not split either — {{CommentSprite}} and {{ProgressSprite}}.

Suggestion Edit

With all above in mind, I offer these things to consider:

  1. Rename all File:spritesheetSprite id.png files to File:Icon id.png or some variant thereof. I don’t think we need arbitrary groupings that were previously enforced by the specific spritesheet files that are now gone. Of course, we would need to resolve name conflicts, when two different icons share the same name across different former spritesheets; for example, if a block has a separate item form, the item icon could probably have priority over the block icon.
  2. Replace Invicon files for items with relevant ItemSprite files. Invicons are 32 pixels in size and ItemSprites are 16 pixels, but with the usage of “nearest neighbor” / “crisp edges” / “pixelated” image rendering mode via a CSS class, 16-pixel images upscale to 32 pixels flawlessly. I know there is some sort of a bug with lighting in inventory slots in-game, but is it really worth to replicate unintended game behavior when the reader would recognize the item in question anyway?
  3. Finally update Module:Schematic to not use Module:Sprite. However, something should possibly be done with Schematic’s naming of icons when reuploading those individually. If done together with proposal 1, the {{BlockGrid}} and {{Layered blueprint}} templates could potentially gain access to a wider set of sprites, as could also Schematic itself.
    •  Amendment Those two templates in question do actually have support for additional spritesheets (and the former is actually named {{Sprite grid}}, but nonetheless, implementing proposal 1 might make it a bit easier to specify icons.
  4. Finally split {{CommentSprite}} and {{ProgressSprite}}.
  5. Evaluate any advantages (or lack thereof) the current SpriteFile module has over a simpler set of templates only using parser functions, and maybe phase the module out.

One of the reasons I am bringing it up right now is because the Russian wiki is going to do away with spritesheets soon as well (for now, we are splitting InvSprites for mods which don’t have English counterparts by definition). The fork has gifted us with a shared file storage with the English wiki, something that we never had ever since joining the larger Minecraft Wiki in early 2011, and even though I opposed using it, there are more users who thought it would be helpful for us. As a result, Russian wiki users deserve to know which English file names could we redirect to.

 Strong Support #2 and #4. Progress Sprite is currently completely broken and I have tempoarily commented out all of it's usage until it can be fixed.
For #1 however, I'm wary of combining all sprite icons into a single namespace, which could potentially make organization more difficult. Right now Minecraft Earth sprites are lumped with vanilla Minecraft, and old item textures share the same namespace with current item textures. When Operation Sprite Ender was completed, the original categorized list of items was replaced by an alphabetical list of items, which makes finding related items harder.
Plus the issue of combining blocksprite and itemsprite into one, if one texture gets the default treatment and the other needs a manual override. I prefer the current system where one must specify "block" or "item" texture in those cases, with no preferred default.
 Support 3, but note that there are some user created spritesheets left over still that use Module:Sprite. Something should be done with the naming issue and the widespread use of those current problematic name IDs.
I do not have any opinion for #5. Delvin4519 (talk) 18:36, 4 December 2023 (UTC)Reply[reply]

Also I  support #4 for obvious reasons.
